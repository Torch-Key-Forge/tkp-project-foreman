# Project Foreman — Three-Minute Demo Script

## Target Duration

2 minutes 35 seconds to 2 minutes 50 seconds.

The remaining time is safety margin. Do not fill it.

---

## 0:00–0:15 — Opening

**Screen:** Project Foreman dashboard before loading the fixture.

**Narration:**

> Long AI conversations often contain real project work: decisions, failures, source references, and unfinished tasks. But raw conversation history is not a durable project workspace.

Pause briefly on the empty interface.

---

## 0:15–0:30 — Scale Proof

**Screen:** Show one clean title card or overlay with:

- 328 conversations
- 29,345 turns
- 52 reconstructed forks
- 180 verified structured operator commands

**Narration:**

> We validated the underlying recovery pipeline against a private corpus containing more than twenty-nine thousand turns. For this demonstration, I am using a small public-safe fixture so the run is fast and predictable.

Do not show the private corpus.

---

## 0:30–0:48 — Load Fixture

**Action:** Click `Load Fixture`.

**Narration:**

> This fixture represents a project that grew through architecture decisions, a detour, a failed path assumption, a repair, an accepted local baseline, and unfinished work.

Wait until the loaded-turn count is visible.

---

## 0:48–1:08 — Recover Project

**Action:** Click `Recover Project`.

**Narration:**

> Project Foreman reconstructs the project into four linked products: a project spine, an authority ledger, a continuation brief, and a source-trace index.

Wait for all metrics and panels to populate.

---

## 1:08–1:30 — Project Spine

**Action:** Click:

1. `Local Prototype Authorized`
2. `Hardcoded Path Failure`
3. `Path Repair Authorized`
4. `Local Baseline Accepted`
5. `Operator Interface Incomplete`

**Narration:**

> The Project Spine compresses the conversation into the events that changed project state. It preserves failures and corrections instead of flattening the history into a generic summary.

Let the source panel update as events are clicked.

---

## 1:30–1:57 — Authority Ledger

**Action:** Click one canonical authorization and one provisional natural-language entry.

**Narration:**

> The authority ledger separates direct operator commands from assistant proposals and ordinary conversational language. Structured commands are canonical. Natural-language candidates remain visibly provisional until reviewed. The system does not silently give the assistant authority.

Pause on the authority status labels.

---

## 1:57–2:22 — Continuation Brief and Trust Demonstration

**Action:** Open `Continuation`. Click the explicit production boundary.

**Narration:**

> The continuation brief shows the last trustworthy state, completed work, unresolved work, active boundaries, and the next recommended action. Here, production deployment remains unauthorized.

Point to the source evidence panel.

> Clicking the claim reveals the exact source turn that supports it.

This is the most important interaction. Do not rush it.

---

## 2:22–2:38 — Export Package

**Action:** Click `Export Package`.

**Narration:**

> The recovered project exports as a real structured package containing Markdown, JSON, a manifest, checksums, and a complete source-trace index.

Show the validated export status.

---

## 2:38–2:50 — Close

**Screen:** Return to the populated dashboard.

**Narration:**

> Project Foreman does not merely summarize a conversation. It reconstructs project state while preserving the evidence needed to verify every claim and continue the work.

Stop.

---

## Recording Rules

- Use the v0.1.2 TKP-branded validated-export build.
- Close unrelated browser tabs and notifications.
- Use browser zoom between 90% and 100%.
- Keep the mouse still during narration.
- Do not explain internal TKP terminology.
- Do not open the private corpus.
- Do not add live improvisation unless the run fails.
- Record at least two complete takes.
- Keep the strongest complete take, not the most enthusiastic partial take.
